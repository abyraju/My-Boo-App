01

App Overview

A private, intimate mobile app built exclusively for couples. Every design decision serves one goal — making the app feel like a warm personal space that belongs only to the two people in the relationship.

🔒
Private by Design

Database-level rules block all third-party access. Only the two paired users can ever read or write their couple space.

💜
Warm & Intimate

Not a social network — a personal sanctuary. Deep velvet backgrounds, soft rose accents, and serif display type create a cozy, romantic feel.

✦
Minimal UI

Every screen is clean and purposeful. No clutter — just the essentials, beautifully presented. Each screen feels like it belongs only to them.

📍
LDR First

Designed for long-distance couples, but valuable for all. Features focus on togetherness regardless of physical distance.

Database Schema (Minimal — Identity & Pairing Only)

With user-owned cloud storage, the central database is minimal — it only handles identity, pairing, and presence. All shared couple data (chat, gallery, etc.) lives in the user's cloud, not here.

users
├── user_id Firebase/Supabase UID
├── boo_id 6-char unique alphanumeric · permanent
├── name, email, phone, country, birthday
├── profile_picture_url string → Firebase/Supabase Storage
├── coupled_with FK → users.user_id · nullable
└── couple_id FK → couples.couple_id · nullable
couples
├── couple_id
├── partner_1_id FK → users.user_id
├── partner_2_id FK → users.user_id
├── couples_key 9-char AES-256 encryption key · generated on pairing · stored in device Keychain/Keystore
└── created_at used for Milestone Tracker countdown
presence (new — for partner activity indicator)
├── user_id FK → users.user_id
├── is_online boolean
├── last_seen timestamp
└── updated_at timestamp

06

App Style

"Velvet Romance" — deeply personal, quietly luxurious, free of social-network energy. Opening the app should feel like opening a private journal, not launching a product.

Background Treatment

All screens use layered dark purple-blacks (#130818 / #0D0511) with radial gradient meshes pushing lighter plum tones toward the center-top. Creates a natural visual focus without visible borders.

Glassmorphism Cards

Cards use rgba(255,255,255,0.05) — barely-there white overlays — with a 1px rgba(255,255,255,0.08) border. Surfaces feel lightweight and part of the background rather than floating objects.

Motion & Floating Hearts

Floating heart decorations animate at four corners of the Sign-In screen using 6–9s ease-in-out vertical floats with slight rotation. Different speeds and phase offsets make them feel alive but never distracting.

Form Design

Inputs have a near-invisible dark fill that brightens subtly on focus with a rose-colored border. Gentle, confident — the input highlights just enough to confirm the user is in it.

Buttons

Security & Sync Architecture

No central database. No developer-hosted storage. Data lives in the user's own cloud account, encrypted end-to-end with a key only the two devices ever hold.

✅

Architecture Decision — User-Owned Cloud Storage

Instead of a central database (Firebase/Supabase), the app uses the user's personal cloud account (Google Drive, Dropbox, OneDrive) as a private encrypted data relay. The developer never hosts, sees, or pays for stored data. Three alternatives were evaluated and rejected: Gun.js (unreliable NAT traversal, weak encryption defaults), Yjs + PeerJS (requires both peers online simultaneously, complex setup).

💰 Zero hosting cost 📈 Infinite scale 🛡 Zero developer liability 📵 Offline-first

🔐 Login Methods

MethodNotes
Google OAuthFast frictionless sign-in · doubles as cloud storage provider
Boo PrivateFull name · country · phone · email · birthday · optional avatar
Apple Sign-In⚠ Mandatory before iOS App Store release

App uses OAuth 2.0 scopes — the user never shares their password with the app. Restricted appDataFolder permission ensures the app can only see its own hidden folder.

🔒 App Lock

After initial login, a secondary lock layer is set up — like a banking app. Users don't re-authenticate from scratch on every open.

MethodDetail
PIN4–6 digit. Set during onboarding. Required fallback when biometrics fail.
FingerprintTouch ID (iOS) · Android Fingerprint API
Face UnlockFace ID (iOS) · Android Face Recognition API
  • PIN stored hashed in device Keychain / Keystore — never plaintext
  • 5 wrong attempts → temporary lockout
  • Danger Zone entry requires PIN explicitly — biometrics not accepted

🔑 The Couple's Key — AES-256 Root

When two users pair, a 9-character Couple's Key is generated. This key is the AES-256 encryption-decryption key for all synced data. It never leaves the device and is never transmitted over the network.

Generated onSuccessful pairing — automatic, no user action
Stored inDevice Keychain (iOS) / Keystore (Android) — locked by PIN / biometrics
Transmitted?Never — the key never leaves the device
VisibilityOnly visible within the app. Must never be shared with a third person.
Cleared onUnpair or account deletion

🔄 Sync Workflow — 4 Steps

💾
1 · Local Save
Save instantly to device IndexedDB. App is fully offline-capable — never blocked by network.
🔐
2 · Encrypt
Scramble with the Couple's Key (AES-256) directly on device. Plaintext never leaves the phone.
☁️
3 · Upload
Push encrypted blob to the user's cloud appDataFolder. Cloud provider sees ciphertext only.
📲
4 · Listen & Decrypt
Partner's device polls / receives push, downloads blob, decrypts locally, merges into IndexedDB.

☁️ Supported Cloud Providers

ProviderAPIScope / Hidden FolderFree
Google DriveDrive API v3drive.appdata15 GB
DropboxAPI v2App folder2 GB
OneDriveMS Graphapproot5 GB

The appDataFolder scope ensures the app is completely blind to the user's other personal files — it can only access its own hidden folder.

🕵️ Metadata Mitigation

Content is fully encrypted, but the cloud provider can still see the envelope (metadata). These strategies minimise what's exposed:

LeakedWhat's VisibleMitigation
IdentityGmail accounts syncingUse dedicated accounts if needed
TimestampsWhen app is activeDummy syncs at random intervals
VolumeFile sizesFixed padding — every file is exactly 1 MB
FilenamesCould reveal contextRandom UUIDs — no descriptive names

⚡ Conflict Resolution

Both devices may write simultaneously. A Last-Write-Wins (LWW) + CRDT-lite merge strategy handles this:

  • Each update carries a timestamp and device ID
  • Same field edited by both → latest timestamp wins
  • Different fields edited → both changes merged cleanly
  • Short version log (last 5–10 states) kept for rollback if needed

🆔 Pairing & Access Security

  • 6-character Boo ID is public-facing — share freely to invite a partner
  • Pairing blocked if target ID is already paired — enforced at DB level, not just UI
  • Home screen only renders after a valid couple_id is confirmed
  • On unpair: Couple's Key cleared from both devices · cloud sync folders wiped
  • Delete Everything: PIN re-entry + type DELETE to confirm · all data purged · both users signed out
